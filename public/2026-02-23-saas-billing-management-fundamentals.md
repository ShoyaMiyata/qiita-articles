## はじめに

SaaS（Software as a Service）ビジネスは、従来の「売り切り型」のソフトウェア販売とは異なり、顧客と継続的な関係を築くサブスクリプションモデルが基本となります。

この記事では、SaaSビジネスの裏側を支えるシステムやルールについて、**「一般的な知識（Level 1）」**として知っておくべき内容を解説します。特に以下の3つのテーマに焦点を当てます：

1. 世の中のSaaS管理システムの機能
2. **世の中のプロダクトの一般的な課金形態**
3. **なぜZuoraやStripeを使うのか？CRMとの違いは？**
4. SaaSの役務提供を裏付ける証憑と法的根拠

## 1. 世の中のSaaS管理システムとは？その主な機能

SaaSビジネスでは、顧客が毎月・毎年継続してサービスを利用するため、Excelや手作業での管理にはすぐに限界が来ます。そこで導入されるのが、**Zuora**、**Stripe Billing**、**Scalebase**などの「**SaaS管理システム（サブスクリプション管理システム）**」です。

一般的なSaaS管理システムは、主に以下のような機能を持っています。

### 1.1 サブスクリプション・ライフサイクル管理

新規契約だけでなく、期中のプラン変更（アップグレード・ダウングレード）、ID数の追加・削減、一時停止（休会）、解約、契約更新といった**「顧客のライフサイクル全体」の履歴を保持し、一元管理**します。

### 1.2 エンタイトルメント（権限付与）管理

「A社はプレミアムプランだから全機能が使える」「B社は50名までログインできる」といった、**契約内容（お金）をプロダクト側が理解できる利用権限（システム設定）に翻訳して連携する司令塔**の役割を果たします。

### 1.3 複雑な請求・ダウニング（督促）の自動化

日割り計算や、段階的な料金プランの計算を自動で行います。また、クレジットカードの有効期限切れ等で決済が失敗した場合に、自動で再請求や顧客への通知を行う**「ダウニング処理」機能**を備えています。

### 1.4 SaaSメトリクス（KPI）の可視化

「いつからいつまで有効か」という期間のデータを持っているため、SaaS経営に不可欠な**ARR（年間経常収益）やMRR（月次経常収益）、チャーンレート（解約率）**といった指標を正確にリアルタイムで算出します。

## 2. 世の中のプロダクトの一般的な課金形態

SaaSの料金体系は、ターゲット顧客やプロダクトの性質に合わせて様々なモデルが存在します。ここでは、**2024年時点で主流となっている課金形態**を詳しく解説します。

### 2.1 ユーザー数課金（Per-User / Seat-based）

利用する人数（ID数やライセンス数）に応じて料金が増加する、**最もポピュラーなモデル**です。

**代表例:**
- Slack
- Zoom
- Salesforce

**メリット:**
- 収益が予測しやすい
- 顧客にとって理解しやすい
- スケールしやすい

**デメリット:**
- 複数人で1アカウントを共有されるリスク
- 企業規模が大きくなると料金が急激に上がる

### 2.2 従量課金（Usage-based / Pay-as-you-go）

システムの利用量（APIのコール回数、通信量、データ保存容量など）に応じて料金が決まるモデルです。

**代表例:**
- AWS（ストレージ、コンピューティング）
- Twilio（SMS送信数）
- SendGrid（メール送信数）

**メリット:**
- 使った分だけ支払うという公平性・透明性
- 小規模からスタートしやすい
- 利用拡大とともに収益が自然に増加

**デメリット:**
- 顧客にとって月額費用が予測しづらい
- 収益の変動が大きい

**2024年のトレンド:**
海外の調査では、**SaaS企業の46%が何らかの形で従量課金モデルを採用**しており、その割合は年々増加しています。

### 2.3 段階的課金（Tiered Pricing / レンジ課金）

「1〜50名までは〇円、51〜100名までは〇円」といったように、数量の範囲（レンジ）や、使える機能のレベル（Basic、Pro、Enterpriseなど）によって階段状に価格が設定されているモデルです。

**代表例:**
- HubSpot（Starter / Professional / Enterprise）
- Asana（Basic / Premium / Business）
- Notion（個人 / チーム / エンタープライズ）

**メリット:**
- 顧客のニーズに合わせた選択肢を提供
- アップセル戦略が立てやすい
- 多様な顧客層にアプローチ可能

**デメリット:**
- プラン設計が複雑
- 境界線付近の顧客が価格に敏感になる

### 2.4 フラットレート（Flat Rate / 定額制）

ユーザー数や利用量に関わらず、月額〇〇円と完全に固定されているモデルです。

**代表例:**
- Netflix
- Spotify
- Adobe Creative Cloud（一部プラン）

**メリット:**
- 顧客にとって分かりやすい
- 請求管理がシンプル
- 予算管理しやすい

**デメリット:**
- ヘビーユーザーからの収益機会を逃す
- スケールしづらい

### 2.5 ハイブリッド型（基本料金 + 従量課金）

**「基本となるプラットフォーム利用料（または最低利用料金）」を設定し、基本枠を超過した分については「ユーザー数」や「利用量」に応じて追加課金**する複合的なモデルです。

**代表例:**
- Snowflake（ストレージ基本料 + クエリ従量課金）
- Datadog（ホスト数 + ログ従量課金）
- GitHub（基本ユーザー + Actions従量課金）

**メリット:**
- 安定した収益ベース + アップサイドの両立
- 顧客の成長に合わせて柔軟に対応
- 最低収益を確保しつつ成長余地を残せる

**デメリット:**
- 料金体系が複雑で説明が難しい
- システム実装が複雑

**2024年のトレンド:**
近年、**ハイブリッドモデル**が急速に増加しています。安定した収益を確保しつつ、利用量増加によるアップサイドも狙えるため、**多くのSaaS企業が採用**しています。

### 2.6 フリーミアム（Freemium）

基本的な機能は無料で使用でき、その上で機能や容量の追加分には課金されるという料金設定です。

**代表例:**
- Slack（無料版 + 有料プラン）
- Zoom（40分制限の無料版 + 有料プラン）
- Dropbox（無料ストレージ + 有料拡張）

**メリット:**
- ユーザー獲得コストが低い
- 口コミで広がりやすい
- 実際に使ってから購入できる安心感

**デメリット:**
- 無料ユーザーのサポートコストがかかる
- 有料転換率は平均3%程度と低い

### 課金形態の選び方

| 課金形態 | 向いているプロダクト | 重視する指標 |
|---|---|---|
| **ユーザー数課金** | コラボレーションツール、CRM | 利用者数が価値に直結 |
| **従量課金** | インフラ、API、通信サービス | 使用量が価値に直結 |
| **段階的課金** | マーケティングツール、プロジェクト管理 | 機能差別化が明確 |
| **フラットレート** | コンテンツ配信、エンタメ | シンプルさ重視 |
| **ハイブリッド** | データ分析、開発ツール | 安定収益 + 成長性 |
| **フリーミアム** | 個人〜中小企業向けツール | ユーザー獲得重視 |

## 3. なぜZuoraやStripeを使うのか？

SaaSビジネスを運営する際、多くの企業が**Zuora**や**Stripe Billing**といったサブスクリプション管理プラットフォームを導入しています。では、なぜこれらのツールが必要なのでしょうか？

### 3.1 自前で実装すると何が大変なのか？

サブスクリプションビジネスを自前のシステムで管理しようとすると、以下のような複雑な処理を実装する必要があります：

#### 1. **日割り計算・期中変更の処理**

「月の途中でプラン変更した場合、どう請求するか？」
- プロレーション（日割り計算）
- 即時変更 vs 次回更新時変更
- アップグレード時の差額請求
- ダウングレード時の返金処理

#### 2. **決済失敗時のリトライとダウニング**

クレジットカードの有効期限切れや残高不足で決済が失敗した場合：
- 自動リトライのスケジュール設定
- 顧客への通知メール送信
- サービス停止のタイミング制御
- 未払い債権の管理

#### 3. **契約更新の自動化**

年間契約や月間契約の自動更新を管理：
- 更新日の自動計算
- 更新前の通知
- 自動更新のキャンセル処理
- 契約履歴の保持

#### 4. **収益認識の複雑さ**

会計基準（企業会計基準第29号）に準拠した処理：
- 前受金の管理
- 期間按分による収益計上
- 解約時の収益調整
- 財務レポートの作成

これらを**すべて自前で実装・運用・保守するコストは膨大**です。そこで、ZuoraやStripe Billingが登場します。

### 3.2 Zuoraを使う理由

Zuoraは、**エンタープライズ向けの包括的なサブスクリプション管理プラットフォーム**です。

#### Zuoraの強み

| 機能 | 詳細 |
|---|---|
| **Quote-to-Cash（見積から回収まで）** | 見積作成、契約管理、請求、回収、収益認識までを一気通貫で管理 |
| **複雑な価格設定** | 段階的課金、従量課金、ハイブリッド課金など複雑なモデルに対応 |
| **収益認識と会計** | 企業会計基準に準拠した自動収益認識機能 |
| **ERPとの統合** | NetSuite、SAP、Oracleなどの基幹システムと連携 |
| **グローバル対応** | 多通貨、多言語、各国の税制対応 |

#### Zuoraが向いている企業

- **中規模〜大企業**
- **複雑なサブスクリプションモデル**を持つ企業
- **収益認識と会計の自動化**が必要な企業
- **CRMやERPとの連携**が重要な企業

**データフロー:**
```
CRM（Salesforce） → Zuora → ERP（会計システム）
```

Zuoraは「**CRMとERPを結びつける組織織（the connective tissue between your CRM and ERP）**」と表現されます。

### 3.3 Stripe Billingを使う理由

Stripe Billingは、**決済処理に強みを持つ開発者フレンドリーなプラットフォーム**です。

#### Stripe Billingの強み

| 機能 | 詳細 |
|---|---|
| **決済処理との統合** | Stripe Paymentsとシームレスに連携 |
| **開発者体験** | 充実したドキュメント、APIが使いやすい |
| **柔軟な価格設定** | 定額、従量課金、段階的課金、ハイブリッドに対応 |
| **グローバル決済** | 135通貨以上、主要な決済手段に対応 |
| **迅速な導入** | コードベースでの柔軟な実装が可能 |

#### Stripe Billingが向いている企業

- **スタートアップ〜中小企業**
- **開発者主導のチーム**
- **シンプルで迅速な導入**を優先
- **決済処理がメイン**の用途

### 3.4 Zuora vs Stripe Billing：どちらを選ぶ？

| 比較項目 | Zuora | Stripe Billing |
|---|---|---|
| **対象企業** | 中〜大企業 | スタートアップ〜中小企業 |
| **価格設定の複雑さ** | 超複雑に対応 | 中程度の複雑さまで |
| **収益認識・会計** | ◎ 非常に強い | △ 基本機能のみ |
| **開発者体験** | ○ 良い | ◎ 非常に良い |
| **導入スピード** | 数ヶ月 | 数週間 |
| **カスタマイズ性** | 設定ベース | コードベース |
| **価格** | 高価 | 比較的安価 |

**選択の目安:**

- **Zuoraを選ぶべき:** 既にCRMやERPがあり、複雑なサブスクリプションモデルを一元管理したい場合
- **Stripe Billingを選ぶべき:** 迅速に決済とサブスクリプションを立ち上げたい、開発者の柔軟性を優先する場合

### 3.5 サブスクリプション管理システムを使うビジネス上の理由

Zuora社のレポートによると、**サブスクリプションエコノミーは過去10年間で435%以上成長**しており、サブスクリプションベースのビジネスは収益成長において製品ベースの企業を**平均5.5倍上回っています**。

また、Harvard Business Reviewの調査によると、**新規顧客の獲得は既存顧客の維持に比べて5〜25倍のコストがかかる**ため、顧客維持が最優先事項です。

サブスクリプション管理システムを使うことで：
- **解約率（チャーン）を削減**
- **アップセル・クロスセルの機会を最大化**
- **顧客ライフタイムバリュー（LTV）を向上**

といった効果が期待できます。

## 4. CRMとサブスクリプション管理システムの違い

SaaS業界では、CRM（Salesforceなど）とサブスクリプション管理システム（ZuoraやStripeなど）が併用されることが一般的です。では、この2つのシステムは何が違うのでしょうか？

### 4.1 CRM（Salesforce）の役割

**CRM = Customer Relationship Management（顧客関係管理）**

CRMは、**顧客情報を一元管理して営業・マーケティング活動を支援するツール**です。

#### CRMの主な機能

| 機能カテゴリ | 詳細 |
|---|---|
| **リード管理** | 見込み顧客の情報管理、ステータス管理 |
| **商談管理** | 営業パイプライン、進捗管理、受注予測 |
| **顧客情報管理** | 連絡先、企業情報、過去のやりとり履歴 |
| **活動管理** | メール、電話、訪問などの営業活動記録 |
| **レポート・分析** | 売上予測、営業KPI、ダッシュボード |

**CRMが答える質問:**
- 「今月の受注見込みはどれくらいか？」
- 「A社の担当者は誰で、最後にいつ連絡を取ったか？」
- 「どの営業担当が最も成績が良いか？」

### 4.2 サブスクリプション管理システム（Zuora）の役割

**サブスクリプション管理 = Subscription Management（定期収益管理）**

サブスクリプション管理システムは、**契約後の「お金の流れ」と「サービス提供」を管理するツール**です。

#### サブスクリプション管理システムの主な機能

| 機能カテゴリ | 詳細 |
|---|---|
| **サブスクリプション管理** | 契約開始、プラン変更、解約、更新の履歴管理 |
| **請求・決済** | 請求書発行、クレジットカード決済、日割り計算 |
| **収益認識** | 会計基準に準拠した収益計上、前受金管理 |
| **エンタイトルメント** | 契約内容に基づく利用権限の設定 |
| **SaaSメトリクス** | ARR、MRR、チャーンレート、LTVの算出 |

**サブスクリプション管理システムが答える質問:**
- 「今月のMRR（月次経常収益）はいくらか？」
- 「A社は来月いくら請求すべきか？」
- 「解約率（チャーンレート）は何%か？」

### 4.3 両システムの関係性：データフローで理解する

CRMとサブスクリプション管理システムは、**役割が異なり、連携して動作**します。

#### データの流れ

```
【営業・マーケティング段階】
CRM（Salesforce）
↓
商談成立・契約締結
↓
【契約・請求・サービス提供段階】
サブスクリプション管理システム（Zuora / Stripe）
↓
【会計・財務段階】
ERP（会計システム）
```

**具体例:**

1. **CRMでの活動（契約前）**
   - 営業担当がSalesforceでリードを管理
   - 見積もりを作成し、商談を進める
   - 契約が成立

2. **サブスクリプション管理システムへデータ連携**
   - 契約情報がZuoraに自動連携
   - Zuoraがサブスクリプションを作成

3. **サブスクリプション管理システムでの処理**
   - 毎月の請求書を自動生成
   - クレジットカード決済を実行
   - プラン変更や解約を処理
   - 収益を期間按分して計上

4. **ERPへデータ連携**
   - 収益データが会計システムに連携
   - 財務レポートを作成

### 4.4 なぜ両方必要なのか？

CRMとサブスクリプション管理システムは、**ビジネスプロセスの異なる段階を担当**しています。

| システム | 担当フェーズ | 主な利用者 | 目的 |
|---|---|---|---|
| **CRM** | 契約前 | 営業、マーケティング | 新規顧客獲得 |
| **サブスクリプション管理** | 契約後 | カスタマーサクセス、財務 | 顧客維持・収益最大化 |

**連携のメリット:**

1. **顧客への適切なアプローチ**
   - 解約リスクが高い顧客を自動で検知
   - カスタマーサクセスがフォローアップ

2. **アップセル・クロスセルの最適化**
   - 利用状況データをCRMに連携
   - 成長している顧客に上位プランを提案

3. **データの一元化**
   - 顧客の全ライフサイクルを可視化
   - 営業からカスタマーサクセスへのスムーズな引き継ぎ

### 4.5 実例：食べログのケース

食べログは、CRMツールとしてSalesforceを使っており、見積、契約管理、請求機能を拡張・強化できる機能を有しているZuoraを導入しました。

**導入理由:**
- Salesforceだけでは複雑なサブスクリプション管理に対応しきれない
- Zuoraと連携することで、販売、契約管理、請求計算・発行、決済、入金・回収管理、売上計上、会計締め処理、分析・レポートまでを自動化

**結果:**
大規模販売管理システムを財務会計SaaSシステムに置き換え、業務効率が大幅に向上しました。

## 5. SaaSの役務提供を裏付ける「証憑」と法的根拠

SaaSビジネスにおいて、「私たちは確かにお客様にサービスを提供しました」という証拠（＝**役務提供証憑**）はどのように定義され、どのような法律が関わっているのでしょうか。

### 5.1 SaaSにおける「役務提供」の考え方

従来のモノの販売であれば「納品書」や「受領書」が証拠になりますが、SaaSのようなクラウドサービスの場合は少し考え方が異なります。

**SaaSの役務提供 = システムをいつでも利用できる状態（Availability）にしたこと**

具体的には：
- アカウントやID/PASSを発行
- アクセス権を付与
- システムが正常稼働している

これらをもって、**顧客が毎日ログインして機能を使ったかどうかに関わらず、役務提供とみなす**のが一般的です。

### 5.2 根拠となりうる主な法律・基準

これらを裏付ける法的・会計的なルールとして、以下のようなものがあります。

#### 1. **民法（契約の基礎と準委任）**

**関連条文:**
- 第521条: 契約の成立（申込みと承諾）
- 第656条: 準委任契約
- 第415条: 債務不履行

**SaaSへの適用:**
- SaaSの利用契約は、申込みと承諾により成立
- SaaSの契約形態は「**準委任契約**」としての性質を持つ（システムの利用環境を提供）
- システムが停止するなどで提供義務が果たされなかった場合、債務不履行による損害賠償の対象となり得る

#### 2. **収益認識に関する会計基準（企業会計基準第29号）**

**概要:**
売上をいつ計上するかというルールです。

**SaaSへの適用:**
- SaaSのように一定期間にわたりサービスを提供する場合、「**履行義務が一定期間にわたり充足される**」とみなされる
- 契約期間に応じて毎月（または日割りで）収益を認識・計上

**実務上の処理:**
```
契約: 年間120万円（12ヶ月契約）
↓
収益認識: 毎月10万円ずつ計上
↓
前受金として管理し、期間按分で収益化
```

#### 3. **消費税法**

**関連条文:**
- 第2条第1項第8号: 役務の提供の定義
- 第6条第1項: 課税の対象

**SaaSへの適用:**
- SaaSのシステム利用権の提供は、消費税法上の「**役務の提供**」に該当
- 課税対象取引となる

#### 4. **電子帳簿保存法**

**関連条文:**
- 第4条: 国税関係帳簿書類の電磁的記録による保存
- 第7条: 電子取引の取引情報に係る電磁的記録の保存

**SaaSへの適用:**
- 契約書や請求書、システム上の利用ログなどを電磁的記録（データ）として保存
- SaaSの管理システム上のデータそのものが、監査に耐えうる証憑として扱われる

### 5.3 実務上の証憑管理

SaaS事業者は、以下のようなデータを証憑として保持・管理します：

| 証憑の種類 | 保存データ | 保存場所 |
|---|---|---|
| **契約証憑** | 利用規約への同意ログ、契約書PDF | サブスクリプション管理システム |
| **役務提供証憑** | システムアクセスログ、稼働率データ | アプリケーションログ、監視システム |
| **請求証憑** | 請求書、領収書 | サブスクリプション管理システム |
| **決済証憑** | クレジットカード決済履歴 | 決済代行サービス（Stripe等） |

これらのデータを**統合的に管理できるのが、ZuoraやStripe Billingの強み**です。

## まとめ

この記事では、SaaSビジネスの基礎となる3つのテーマを解説しました。

### 重要ポイント

1. **SaaS管理システムの役割**
   - サブスクリプション・ライフサイクル管理
   - 複雑な請求・決済の自動化
   - SaaSメトリクスの可視化

2. **一般的な課金形態**
   - ユーザー数課金、従量課金、段階的課金、フラットレート、ハイブリッド、フリーミアム
   - **2024年のトレンド:** ハイブリッドモデルと従量課金が増加中

3. **ZuoraやStripeを使う理由**
   - 複雑な処理（日割り、期中変更、リトライ、収益認識）を自前で実装するコストが膨大
   - **Zuora:** エンタープライズ向け、Quote-to-Cash全体をカバー
   - **Stripe Billing:** スタートアップ〜中小企業向け、決済処理に強み

4. **CRMとの違い**
   - **CRM（Salesforce）:** 契約前の営業・マーケティング活動を管理
   - **サブスクリプション管理（Zuora/Stripe）:** 契約後の請求・決済・収益管理を担当
   - **データフロー:** CRM → サブスクリプション管理 → ERP

5. **法的根拠**
   - 民法（準委任契約）
   - 収益認識に関する会計基準
   - 消費税法（役務の提供）
   - 電子帳簿保存法

### 次のステップ

このLevel 1の知識を土台に、次は以下のような「Level 2」の理解を目指しましょう：

- **自社のSaaS管理機能の具体的な設計**
- **自社プロダクトの規約と役務提供内容の整理**
- **自社の正価表とプライシング戦略の策定**

SaaSビジネスは、単なるソフトウェア販売ではなく、**顧客と長期的な関係を築くビジネスモデル**です。その土台となるシステムと法的理解をしっかり押さえることが、事業成功の第一歩となります。
